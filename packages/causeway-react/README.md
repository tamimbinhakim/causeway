# @causewayjs/react

React hooks for the Causeway route-key client. One provider, one query hook, one mutation hook.

```bash
pnpm add @causewayjs/react @causewayjs/client
```

Causeway has one React surface:

```tsx
import { useMutation, useQuery } from "@causewayjs/react";

const customer = useQuery("GET /customers/$id", { id });

const screen = useMutation("POST /customers/$id/screen", {
  feedback: {
    loading: "Screening customer...",
    success: "Customer screened",
  },
});

await screen({ id });
```

## Setup

```tsx
import { createClient } from "./client"; // generated by causeway
import { CausewayProvider } from "@causewayjs/react";

const client = createClient({ baseUrl: "/api" });

export function Providers({ children }: { children: React.ReactNode }) {
  return <CausewayProvider client={client}>{children}</CausewayProvider>;
}
```

Feedback is app-owned, so Causeway does not choose your toast library:

```tsx
<CausewayProvider
  client={client}
  feedback={{
    loading: (message, id) => toast.loading(message, { id }),
    success: (message, id) => toast.success(message, { id }),
    error: (message, id) => toast.error(message, { id }),
  }}
>
  {children}
</CausewayProvider>
```

## Hooks

`useQuery(routeKey, input, options)` returns:

- `data`
- `pending`
- `error`
- `refresh()`
- `setData(next)`

`useMutation(routeKey, options)` returns a callable mutation with:

- `pending`
- `error`
- `data`
- `reset()`

Import the generated client once in your app setup and `@causewayjs/react`
infers `input`, `data`, and `error` from the route key. No
`useQuery<Customer>(...)` ceremony in normal app code. Runtime caching,
in-flight dedupe, mutation state, hydration, and declared `refreshes` live in
`@causewayjs/client`.

There is no TanStack Query dependency in the primary runtime. Causeway keeps the cache semantics small and route-key-native so React Compiler-era apps do not have to wrap every input object in ceremony.
